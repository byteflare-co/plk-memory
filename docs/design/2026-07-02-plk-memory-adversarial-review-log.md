### [high | valid | must-fix] Graphiti の差別化価値に成功指標がない — 非グラフ・ベースライン比較の欠如

**視点**: false-green, data-quality

**指摘内容**: §11 Phase 1 の完了条件は precision@5・ファクト単価・利用記録のみで、§13 の比較計画も embedder 3種（すべて Graphiti 内部の差し替え）に閉じている。同じ markdown コーパスに対する非グラフ検索（grep/FTS5/素の埋め込み検索。§8 で縮退読み口として既に存在する）との対照比較がない。また 組織展開 設計が Graphiti を選んだ理由である矛盾検出・時間推論は評価計画に一度も登場せず、ソロ数十件のコーパスでは矛盾は自然発生もしない。

**なぜ問題か**: このままだと「検索が当たる・コストが許容」で緑になり得るが、その緑は「markdown＋素の埋め込み検索でも同じ結果だった」と識別不能。組織展開 が本当に知りたい「エピソードあたり十数回の LLM 呼び出しとグラフ DB 運用を払う価値があるか」に答えないまま「Byteflare で実運用検証済み」が独り歩きする。1人法人の小コーパスはむしろベースラインが勝ちやすい局面で、そこを測らないのは false green の温床。

**提案された修正**: Phase 1 評価に同一 20 クエリを (a) plk_search と (b) markdown への素の埋め込み/FTS 検索で比較する対照群を必須化する。矛盾検出は人工的な矛盾系列（同一トピックの update 連鎖）をシードに仕込み、週次キュレーションの検出率を測る項目を追加する。ベースラインに勝てなかった場合はその事実を逆輸入レポートの一級知見として明記し、撤退ライン判断（グラフ層凍結）に直結させる。

**検証者の判定**: 指摘の事実主張はすべてドラフトの記述で裏付けられ、反駁は成立しなかった。(1) §11 Phase 1 完了条件（186行目）は precision@5・ファクト単価・利用記録の3点のみで対照群がなく、§13（205行目）の比較も Voyage/Gemini/OpenAI という Graphiti 内部の embedder 差し替え3種に閉じている。(2) 非グラフ読み口（grep/markdown）は §8（154行目）に縮退経路として既存であり、§1（17行目）で Track A 索引に PLK markdown を含める決定まであるため、ベースライン対照はほぼ無料で実施可能なのに評価計画に不在。(3) 矛盾検出は §9 週次キュレーション（160行目）に運用機能として登場するだけで検出率を測る項目が §11/§13/§14 のどこにもなく、時間推論は plk_history が「git log＋superseded_by チェーン」（117行目）と Git 側実装のため Graphiti の bi-temporal は評価どころか利用経路すら薄い。(4) v3 §8.3（164行目）は「矛盾を仕込んだ系列で測らないと機能が遊ぶ」と明示警告しており、「v3 の教訓は継承する」（ドラフト §1、13行目）という自己宣言と矛盾する脱落。(5) §10 の「検証されない」リスト（174〜179行目）にも「対ベースラインの差別化価値」が含まれず、false-green 防壁自体に穴がある。反駁として「技術選定妥当性は 組織展開 側 PoC の責務」を検討したが、§10 にその切り分けが明記されていない以上、撤退ライン（191行目）が絶対閾値のみで判定され「markdown＋FTS でも同等」と識別不能な緑が「実運用検証済み」として逆輸入される経路が現に開いており、棄却。「小コーパスでベースラインが勝っても事業規模への反証にならない」という反論は suggested_fix の撤退直結部分に部分的に刺さるが、指摘の中核（対照群なしでは緑が解釈不能）には刺さらない。materiality: 本パイロットのコストの大半（十数 LLM 呼び出し/ファクト＋グラフ DB 運用）はグラフ層に集中し、その価値識別がチェックポイント判断の核心であるにもかかわらず評価設計が構造的にそれに答えられない。修正コストは低く（同一 20 クエリを既存 grep/FTS/素の埋め込みに流す＋update 連鎖の人工矛盾シード）、Phase 1 開始前に組み込むべきため must-fix。

### [high | partially-valid | must-fix] 「自 namespace」概念が Byteflare の実態に写像できず、デフォルト検索と write 認可が定義不能

**視点**: ops-yagni, data-quality, false-green

**指摘内容**: §3 は namespace をドメイン別ディレクトリ（tax/legal/shaho/dev/backoffice/biz）で切り、§5 は認可を「write:自ns」、デフォルト検索を「shared＋自 namespace の 2 group」と定めている。しかし 組織展開 では namespace＝部署＝エージェントの所属で 1:1 に決まるのに対し、Byteflare の主要クライアント（Claude Code / Codex / Hermes）は全員が汎用で、同じ日に tax も dev も backoffice も扱う。つまり CC にとっての「自 namespace」が存在しない。

**なぜ問題か**: 実害は日常利用の中核で出る。(a) デフォルト検索「shared＋自ns」が決められないため、毎クエリで namespace 指定を強いるか、6 domain 全部への fan-out になる——後者は §3 自身が検証済みファクトとして禁じた「FalkorDB は group 横断で fan-out コスト増、デフォルトは 2 group まで」と正面衝突する。(b) 「社保と税務どちらの扱いか」のような domain 横断の質問（1人法人では頻出）がデフォルトで引けない。(c) write:自ns の認可はトークンに namespace を紐づけない限り執行不能で、紐づけると CC に 6 トークン必要という不条理になる。結果、検索が当たらない→使わない→precision@5 も悪化、という「作って使わない」への直行ルートになる。

**提案された修正**: Byteflare モードでは group_id を 1 個（せいぜい shared/staging の 2 個）に畳み、namespace は frontmatter のメタデータ（検索フィルタ）としてのみ保持する。group_id マッピングを設定可能（Byteflare: 全 namespace→単一 group、組織展開: namespace→group 1:1）にすれば逆輸入互換は保てるし、FalkorDB fan-out 問題も消える。認可は read / write / admin の 3 種に簡略化し、「write:自ns」は 組織展開 側 PoC で検証する項目として §10 の『検証されないもの』に移す。

**検証者の判定**: 核心は反駁できない。ドラフト §3（L57-58）は namespace=ディレクトリを Graphiti group_id に 1:1 マッピングし「デフォルト検索は shared＋自 namespace の 2 group」と定め、§5（L113-116）も plk_search/plk_add/plk_propose_promotion/plk_invalidate の認可を「自 namespace」概念で定義するが、クライアントと namespace の対応規則は文書のどこにも定義されていない。一方 §3 図・§9 接続テンプレート・§11 Phase 2 完了条件が示すとおり、パイロットの一次クライアントは CC/Codex/Hermes/自作1体で全員汎用であり、組織展開 の「所属部署=namespace」前提が写像できないのは事実。つまり主要クライアント全員についてデフォルト検索のセマンティクスが未定義で、Phase 1 の完了条件（日本語 precision@5、「自分が日常で引いたか」）はまさにこの検索デフォルトに依存するため、未定義のまま実装するとパイロットの中核計測が歪み、ドラフト自身が「最恐」と置くリスク「作って使わない」（L210）に直結する。よってこの部分は valid で、Phase 1 着手前に §3/§5 で Byteflare モードの group マッピング（指摘の suggested_fix か、per-call namespace 指定＋2-group 指針の緩和）を決めなければならない = must-fix。ただし以下のサブ主張は反駁できた。(1)「write 認可が定義不能」の実害: セキュリティ上重要な不変条件は §7 L145「shared への直接書き込みは全段階で禁止（昇格 PR 経由のみ）」として namespace-トークン紐づけなしに独立に定義・執行可能で、「write:自ns」は書き込み先 != shared の検証に退化させれば足りる。1人法人では全クライアントが同一人物の代理（単一信頼ドメイン）なので執行不能の実害はほぼなく、「CC に 6 トークン必要という不条理」は誰も選ばない偽の二択。(2)「全 group fan-out は §3 の検証済みファクトと正面衝突だから不可」も過大: L58 の 2-group 制約は fan-out＋マージという機構の指摘から導いた自作の指針であり、数百ファクト・7 group 規模では実測コストはほぼ無視できる（ingest LLM コストは group 数と無関係）。Byteflare モードで指針側を緩めて全 group をデフォルトにする安価な代替解が存在し、指摘はこれを不当に排除している。以上より、指摘の中心命題（自 namespace が写像不能でデフォルト検索が未定義、domain 横断質問がデフォルトで引けない）は valid かつ設計確定前に必須の修正だが、write 認可の執行不能の実害と fan-out 衝突の不可避性は過大表現のため partially-valid とする。

### [high | valid | must-fix] 「作って使わない（最恐）」対策に需要側の動線がなく、v3 の教訓（検索を呼ぶ動線が前提条件）を落としている

**視点**: ops-yagni

**指摘内容**: §13 リスク表は最恐リスク「作って使わない」の対策を「利用ログで週次可視化＋撤退ライン＋シードに実データ」としているが、これらは全て測定側で、エージェントに plk_search を呼ばせる仕掛け（CLAUDE.md / システムプロンプトへの1行、スキルトリガー、フック）が成果物のどこにもない。v3 §7 は「ユーザーが検索を呼ぶ動線があること」を『これが無いと理論上だけになる』3 前提の1つとして明記していたのに、本設計は接続テンプレート（§9）に登録コマンドしか書いていない。

**なぜ問題か**: エージェントは登録された MCP ツールを自発的には引かない（既存の Notion MCP がまさにそうであるように、プロンプト側の指示がなければ呼ばれない）。動線なしだと利用ログは正しくゼロ近辺を示し、Phase 1 チェックポイントで『使われなかった』という結論が出るが、それは基盤の欠陥ではなく配線漏れの偽陰性。逆に人間が『測り方が不公平だった』と感じれば撤退ラインが無効化され、最恐リスクの安全弁が両方向に壊れる。

**提案された修正**: Phase 1 の成果物に呼び出し動線を明示的に含める：各クライアントの設定/スキルに「税務・社保・法務・過去の決定に関わる判断の前に plk_search を引く」の 1 行を入れ、利用ログに『自発（プロンプト誘導）呼び出しか、人間の明示指示か』の区別を記録する。チェックポイント判定は動線導入後の期間のみを対象にする。

**検証者の判定**: 反駁不成立。ドラフトを実際に確認すると、(1) §13 の「作って使わない（最恐）」対策（L210）は利用ログ可視化・撤退ライン・シード選定のみで全て測定側/供給側、(2) §9 接続テンプレート（L162-166）は登録コマンドとツール名プレフィックスのみ、§8（L152）でテンプレートに入るのは「タイムアウト→メモリなしで続行」という使わない方向の契約だけで、plk_search を引かせるプロンプト側の1行・スキル・フックは §11 の全 Phase 成果物を含めどこにもない。(3) v3 §7（L134）は「検索を呼ぶ動線（コマンド or システムプロンプト1行）」を『これが無いと理論上だけになる』3前提の1つと明記しており、ドラフトは §9 で「v3 チェックポイント思想」を引用して測定側だけ継承し動線を落としている——指摘の構図は全て正確。反駁候補も検討したが棄却: ①「1人パイロットは人間が動線」→ §1 の目的は「エージェント群が読み書きする基盤」であり、テンプレートは §10 で 組織展開 へ持ち上がる成果物なので動線欠落ごと輸出される。②「シード実データ選定が需要側対策」→ 引く価値と引く動線は別問題。③「MCP ツールは description 次第で自発呼び出しされ得る」→ ドラフトはトリガーを意識した description 設計も規定していない。唯一の誇張は「撤退ラインが偽陰性で発動」の部分で、§11 の撤退ライン明文基準（実働2週間/precision@5/ファクト単価）に利用率は含まれないが、チェックポイントの判断入力に「自分が日常で引いたかの記録」（L186）が含まれるため継続/撤退判断が汚染される本質は変わらない。materiality は must-fix: 設計自身が「最恐」と位置づけたリスクへの対策が、パイロットの意思決定ゲート（Phase 1 チェックポイント）と逆輸入成果物の両方で機能不全になる一方、修正はテンプレート1行＋利用ログに呼び出し元（自発/人間指示）の区別を足すだけと安価で、直す理由が明確に勝る。

### [high | valid | should-fix] 「永続層は git-primary → graph-primary のアダプタ差し替え」（§2表・§10）は過大評価で、graph-primary は実質書き直しになる

**視点**: org-port, false-green

**指摘内容**: §2 の確定判断と §10 は「永続層（git-primary → graph-primary オプション）」をアダプタ差し替えで持ち上がる項目に分類しているが、設計本文のどこにもストレージのポート（インターフェース）定義がなく、graph-primary モードはどの Phase でも実装・検証されない。実際には中核機能が全て Git 形状で定義されている: 書き込みパス＝commit が SoT 確定点（§6）、plk_history＝git log＋superseded_by（§5）、監査＝commit 履歴＋追記ログの二重（§2）、ハードデリート＝git filter-repo（§7、graph-primary での代替解なし＝v3 §9.2④の「bi-temporal で消せない」問題が未解決のまま復活）、人間編集の取り込み＝GitHub webhook（§6）。さらに plk_search の kind/status/期間フィルタの実現方法が未規定で、エピソード名 {id}@{content_hash} から SoT ファイルの frontmatter へ逆引きする構造なら読み取りパスまで GitStore に依存する。

**なぜ問題か**: 逆輸入の目的は「コードごと 組織展開 に持ち上げる」こと。組織展開 の参照アーキテクチャは Graphiti ベースの中央ストア（graph-primary）なので、逆輸入時に graph-primary を選んだ瞬間、書き込み・履歴・監査・削除・検索フィルタの各パスを新規実装することになり、「叩き台として丸ごと持ち上がる」という §2 の根拠が崩れる。これは v3 が最恐と警告した false green（Byteflare で快適に動いた＝組織展開 に持ち込めると読み替える）の再演。

**提案された修正**: 二択を明示的に決める。(a) graph-primary を本気でオプションにするなら、FactStore/HistoryProvider/検索メタデータ供給のポート定義を Phase 1 の骨格に前倒しし（実装は Git 版のみでよい）、Phase 3 完了条件に graph-primary のスモーク実装（plk_add/search/invalidate が graph だけで一周する）を含める。(b) それをやらないなら、§10 の「アダプタ差し替えで持ち上がる」から永続層を外し「このパイロットでは検証されない」リストへ移した上で、組織展開 へは git-primary 構成（知識リポジトリ＋派生索引）自体を提案として売り込む方針に倒す。どちらでもよいが、現状の「オプションにできる（未定義・未検証）」という中間状態が最も危険。

**検証者の判定**: 反駁を試みたが核心は崩せなかった。(1) ポート未定義は事実: ドラフト§2(L25)の「バックエンドはプラガブル」は一語の意図表明のみで、FactStore等のインターフェース仕様は本文に不存在。シークレット検知には「スキャナはインターフェースで抽象化」(L71)と明示しており、著者の流儀に照らしても永続層だけ未規定なのは明白。(2) 全Phase未検証も事実: Phase 3の完了条件は「半日でstgに立てられる」(L189)のみで、graph-primaryでの一周動作はどこにも含まれない。(3) 中核パスのGit形状も事実: 書き込み＝commitがSoT確定点（§6 L129、ingest失敗時も「SoTは既に確定」という整合性モデル自体がGit前提）、plk_history＝git log＋superseded_by（L117）、ハードデリート＝git filter-repoを自ら「v3 §9.2④のGit側解」と明記（L147、graph-primary代替解なし）、人間編集取り込み＝GitHub webhook（L131-133）、昇格＝ファイル移動PR（L159）。plk_searchのkind/status/期間フィルタ（L113）の実現方法も未規定で、frontmatter逆引きなら読み取りまでGitStore依存という条件付き指摘も妥当。(4) 実害なし論も棄却: ドラフトは§1(L13)で「何をde-riskしないかを§10で正直に固定する」ことをv3継承の中核に据えており、§10の誤分類はv3が最恐と警告したfalse green(v3 L159)の再演そのもの。しかも「graph-primaryモードを選択可能に」は§2のユーザー確認済み確定判断の根拠列に入っており、未定義・未検証の約束が確定事項に混入している。部分的な過大評価は2点のみ: 監査は§3/§7の追記専用ログが永続層から独立しておりgraph-primaryでもそのまま持ち上がる（消えるのはGitHub側の差分・レビュー監査のみで「監査を新規実装」は半分過大）。「実質書き直し」もシステム全体ではなく永続層隣接パス（書き込み・履歴・削除・昇格・編集取り込み）に限られ、§10の「そのまま持ち上がる」リストは依然正しい。ただしこれらは枝葉で、提案された修正(a)/(b)の二択の妥当性を変えないため valid とする。materialityはshould-fix: Phase 0〜2の実行は影響を受けず、(b)に倒せば逆輸入価値も保存されるためmust-fixではないが、逆輸入は本パイロットの目的の半分であり§10は設計自身が誠実性担保に指定した箇所。(a)を選ぶならPhase 1の骨格に影響するため、着手前に二択を決めるべきで、noteに落とすには重い。

**視点**: false-green, ops-yagni

### [high | valid | should-fix] 非同期 ingest キューと dirty フラグの永続性が未定義 — commit 成功→プロセス再起動で索引乖離が恒久化し plk_status も嘘をつく

### [high | partially-valid | should-fix] 「commit=SoT 確定」と PR 昇格フローが矛盾 — SoT の実体（サーバー clone か GitHub リモートか）が未定義で、push 失敗・non-fast-forward の解消ポリシーがない

**視点**: consistency

**指摘内容**: §6-2 は「git commit（＝SoT 確定）→ push（非同期可）」とするが、SoT のコピーはサーバー clone・GitHub リモート・人間のローカル clone の3つ存在する。人間が main に直接 push（1人法人なら日常的にやる）した状態でサーバーに未 push commit があると non-fast-forward になり、その解消手順（rebase か fail か）が設計に無い。また plk_propose_promotion の PR 作成は push 成功が前提で、未 push commit が溜まった状態でブランチを切ると無関係なファクトが PR diff に混入する。

**なぜ問題か**: 昇格承認・監査・レビュー UI はすべて GitHub 上で回る設計なので、実質の SoT はリモート main。しかし push が非同期・失敗時ポリシー無しだと、「SoT 確定済み」のファクトが EC2 のディスク（耐久性の契約が無い場所）にしか存在しない期間が無限に伸び得る。分岐が起きた瞬間に単一 writer 前提が「clone 間の多重 writer」問題にすり替わり、これは asyncio.Lock では一切守れない。

**提案された修正**: 「SoT＝リモート main、ローカル commit は push 完了までは耐久化された意図に過ぎない」と定義を明文化。書き込みは fetch → rebase → push のリトライループを writer ロック内で回し、コンフリクト時は操作をエラーとして呼び出し元に返す（黙って分岐させない）。plk_status に「未 push commit 数」を必須項目として追加。昇格 PR 作成前に push 完了（ローカル==origin/main）をプリコンディションにする。

**検証者の判定**: 【検証で確認できた事実（ドラフト本文）】§6-2 は「git commit（＝SoT 確定）→ push（非同期可）」とし、push の失敗ポリシーは本当にどこにもない。§6-3 のリトライ＋dirty フラグ＋plk_status 可視化は ingest 専用で、push には適用されていない。§6-4 は EC2 期に「webhook → pull → 差分 ingest」を明記しており、リモート main がサーバー外（人間の PR merge・直接編集）で進むことを設計自身が前提にしている。にもかかわらず、サーバーに未 push commit がある状態での pull（divergent history）の解消方針は未記載。近年の git 既定では pull.rebase 未設定の divergent pull は fatal エラーになるので、Phase 2 の完了条件「昇格が実運用で1回以上回る」経路（merge → webhook → pull）が、直近の plk_add の push 未完と重なった瞬間に手動介入が要る。§13 リスク表にもこの項目はなく、CORS ヘッダや webhook 署名検証まで書き込む本ドキュメントの詳細度からすると明確な仕様欠落。suggested_fix（writer ロック内 fetch→rebase→push、plk_status に未 push 数、PR 前プリコンディション）は安価で妥当。ここまでは valid。

【反駁できた部分】(1)「SoT の実体が未定義」は不正確。§3 の図は GitStore（サーバー側 clone）に明示的に (SoT) と付しており、GitHub リモートは push/pull 先として別記されている。未定義なのは SoT の所在ではなく「分岐時の照合ポリシー」。(2)「昇格承認・監査・レビュー UI はすべて GitHub 上で回る」も不正確。監査ログは §3/§7 でサーバー側の追記専用コンポーネントであり、読み口（plk_search・Web UI・grep）も全てサーバー clone を参照する。GitHub は承認面＋耐久化層であって「実質の SoT はリモート main」は言い過ぎ。(3)「分岐した瞬間に clone 間の多重 writer 問題にすり替わる」は git のセマンティクスを誇張。non-fast-forward push は拒否されるだけで黙った lost update は起きず、1ファクト1ファイル＋ULID 採番（§4）のため実テキストコンフリクトは plk_invalidate と昇格 move の競合程度に限られる。障害の実態はデータ破壊ではなく「同期停止＋操作エラー」という可用性問題で、1人法人なら数分で手動復旧できる。(4)「PR 作成は push 成功が前提で、未 push commit があると無関係ファクトが diff に混入」は半分だけ正しい。新規ブランチの push は main の分岐と無関係にほぼ必ず成功する（新 ref は自明に fast-forward）ので「push 成功が前提」で詰まる主張は弱い。diff 混入はブランチをローカル HEAD から切った場合のみ起き、origin/main から切れば回避可能な実装詳細。混入しても中身は既に commit 済みで main 行きが確定したファクトでありレビューノイズに留まる。(5)「EC2 ディスクにしか存在しない期間が無限に伸び得る」は、push 失敗が完全に無監視である場合の理論値。実際は昇格ブランチ push の失敗や merge 後 ingest の不発（昇格したはずの知見が検索に出ない）で早晩露見する。よって severity high は過大で、should-fix 相当の仕様欠落。

【materiality 判断】1人法人パイロット＋組織展開 逆輸入という目的に対し、この経路は Phase 2 の完了条件そのものであり必ず踏む。かつ「PR 昇格フロー」は逆輸入成果物（§10 でアダプタ差し替えで持ち上がる範囲）に含まれるため、欠落を残すと 組織展開 側にそのまま輸出される。修正コストは数行のポリシー追記で済むため should-fix。must-fix としないのは、失敗が git により loud に顕在化し（黙った消失がない）、単独開発者の手動復旧で回復可能で、パイロットの成否自体を左右しないため。

### [high | partially-valid | should-fix] Phase 1 でサーバーの作業 clone と人間の日常編集ディレクトリが同一になる — asyncio.Lock の外から作業ツリーが書き換わる

**視点**: consistency

**指摘内容**: §3 はデータリポジトリを「agent-organization の knowledge/（7/2 決定の場所をそのまま使用）」とする。agent-organization は西川が日常的にエディタで開き、git checkout / rebase / stash / commit -A を打つ現用リポジトリであり、Phase 1（ローカル）では plk-memory-api が同じ作業ツリーを writer として使うことになる。人間のエディタ保存や git 操作は asyncio.Lock の管轄外で、API の「ファイル書き込み→commit」シーケンスの途中に割り込める（例: 人間の `git commit -A` が API の書きかけファイルを巻き込む、branch 切替で作業ツリーごと入れ替わる）。§9 の週次キュレーションも cron/launchd の別プロセスであり、リポジトリに触るなら同じ問題を踏む。

**なぜ問題か**: 「単一 writer（プロセス内ロック）」という §3 の中核不変条件が、この文脈で最も高頻度の書き込み主体＝本人の日常 git 操作に対して最初から成立していない。1 fact 1 commit・監査ログ・ingest 差分検出のすべてが「作業ツリーは API だけが触る」前提に依存しているため、壊れ方が静かで検出しづらい。

**提案された修正**: サーバーは専用 bare-ish clone（例: ~/.plk/data-repo）を持ち、人間の編集は必ず GitHub（PR or main への push）経由でのみ取り込む構成に変更。ローカル期でも「人間の clone」と「サーバーの clone」を物理的に分ける。週次キュレーションはリポジトリ直触り禁止・API 経由のみと明記。

**検証者の判定**: 核心は反駁不能: ドラフト §3 L56（agent-organization の knowledge/ を「そのまま使用」）＋ §6 L133（ローカル期は人間の直接編集を pull なしの mtime/hash 差分で取り込む）は、Phase 1 で API と人間が同一作業ツリーを共有する構成を含意し、設計には git 状態レベルの干渉への手当て（専用 clone・commit 前の HEAD==main 検証・昇格ブランチの worktree 分離・index.lock 競合処理）が一切ない。§9 の plk_propose_promotion 自体がブランチを作るため共有ツリーでは人間不在でも危険で、常駐 Hermes が任意時刻に plk_add を打てる以上「ブランチ checkout 中に書き込みが来ない」保証もない。ブランチ誤着地・rebase 中 detached HEAD への commit 孤児化は Git=SoT を静かに壊す実在の穴。「設計が既に手当て済み」との反駁は不成立（§4 の CI と §6 の差分 ingest は内容レベルの手当てであり git 状態レベルではない）。一方、指摘には過大・誤認が複数ある: (1) agent-organization は現在 git repo ですらなく（.git なし・knowledge/ 未作成・tmp/ の PDF のみ）「日常的に rebase/stash を打つ現用リポジトリ」「最高頻度の書き込み主体」は事実と不一致の推測、(2) 「ingest 差分検出が API 専有前提に依存」は逆で、mtime/hash JIT はまさに人間編集を吸収する機構、(3) 監査ログは repo 外の追記専用ストアで git 操作では壊れない、(4) commit -A 巻き込み例はロック内サブ秒の窓を要し非現実的、(5) 週次キュレーションは「確定操作は人間」でレポート生成のみ・repo 書き込みの記載なし（条件付き推測）。materiality: Phase 2（EC2 webhook→pull）と 組織展開 逆輸入形は必然的にサーバー clone 分離となり成果物は無傷だが、Phase 1 は継続/撤退判断の計測フェーズであり静かな SoT 汚損は判断を歪める。修正は「サーバー専用 clone（例 ~/.plk/data-repo）＋昇格ブランチは git worktree＋commit 前 HEAD 検証」の明記のみで極小コスト。must-fix ではなく should-fix。

### [high | partially-valid | should-fix] ingest ジョブが edge-triggered（内容を運ぶ）だと、リトライ×順序逆転で重複 episode・古い内容への巻き戻りが起きる — episode 名決定論だけでは冪等にならない

### [medium | partially-valid | should-fix] 昇格＝ファイル移動の episode 同一性が未設計 — group を跨ぐ delete/add の非原子性、rename 検出漏れによる旧 namespace への stale episode 残留、plk_history の履歴断絶

### [medium] force-push / 履歴書き換え後のサーバー clone 追従が未定義 — 自前のハードデリート runbook（git filter-repo）自体がこの穴を踏む

**なぜ問題か**: 静的 Bearer は漏洩に気づいた時しか失効されない。PLK_TOKEN は ~/.zshrc 等の環境変数として Mac 上の任意プロセスから読める上、dotfiles 同期や設定ファイル共有（.mcp.json）でうっかり外に出やすい種類の値。失効リストは「漏れたと分かった後」の道具であり、「漏れたことに気づかない」ケースには期限しか効かない。

**提案された修正**: トークンに発行日と有効期限を持たせ（サーバー側 DB で管理する不透明トークンで十分）、90日程度の強制ローテーションを運用に組み込む。クライアント別トークンは Phase 1 から発行する（指摘1の fix と同じ変更で済む）。

### [low] 監査ログの改竄耐性（ハッシュチェーン・箱外退避）が v3 から脱落している

**視点**: security

**指摘内容**: v3 付録A は監査ログを「追記専用＋ハッシュチェーン＋定期的に箱外へ退避（侵害時に履歴を書き換えられないように）」と定義していたが、本設計 §3/§7 は「追記専用（SQLite or JSONL）・API から到達不可」のみ。同一プロセス・同一ホストが侵害されればファイルごと書き換え可能で、既存の小規模 EC2 は他サービス相乗りのため侵害面を共有している。

**なぜ問題か**: 監査ログの存在意義は「エージェントが何を書いたか」を事後に信頼できることで、記憶汚染インシデントの調査線でもある。侵害シナリオでこそ必要なログが侵害で消せる設計は目的と矛盾する。逆輸入時に 組織展開 の監査要件（§11 系）を満たす形にしておく価値も高い。

**提案された修正**: エントリにハッシュチェーンを付け、S3（既存の小規模 EC2 は AWS 管轄下なので自然）へオブジェクトロック付きで日次退避する 1 ジョブを Phase 2 の完了条件に追加する。実装は小さく、逆輸入時はそのまま CloudTrail 的な監査証跡の叩き台になる。

### [low] 高コスト・高不確実の add_episode がデフォルトで、決定論・ゼロコストの add_triplet がフォールバックという逆転

**視点**: ops-yagni

**指摘内容**: §4 は add_episode（LLM 抽出）を常用経路にし、add_triplet 直挿入を「コストが高すぎた場合のフォールバック」に置く。しかし設計自身が引用する検証済みファクト（エピソードあたり十数回の LLM 呼び出し、日本語 MinHash dedupe 不全で増幅、プロンプト英語のみ）は、この経路が日本語ソロ運用で最も高コスト・最も品質不確実であることを示している。しかも投入するファクトは既に statement/why/how_to_apply に構造化済みで、LLM 抽出の付加価値がそもそも薄い入力形式。

**なぜ問題か**: 常用経路の選択を間違えると、Phase 1 の LLM 費用と ingest 遅延が日々の摩擦になり、撤退ライン（ファクト単価）に不必要に早く到達する。add_episode の実測値が逆輸入資産として必要なのは事実だが、それは常用することとは別の話。

**提案された修正**: 常用 ingest は add_triplet（frontmatter → 明示トリプレット、決定論・LLM 不要）をデフォルトにし、add_episode は『シード20件に対する一回きりの計測実験』として実施して単価・品質データ（逆輸入資産）だけ取る。検索品質評価で LLM 抽出構築のグラフが triplet 構築に明確に勝る場合のみデフォルトを昇格する、と Phase 1 計画に書く。

### [low] add_triplet フォールバックモードの topic の出どころが未定義（tags 任意のままでは退化グラフになる）

**視点**: data-quality

**指摘内容**: §4 の対策④は『frontmatter の構造を (fact)-[RELATES_TO]->(topic) 系の明示トリプレットに変換』とするが、topic に相当するフィールドが規約に存在しない。tags は必須でなくデフォルト空配列のため、フォールバック発動時に生成されるグラフは孤立ノードの集合に退化する。

**なぜ問題か**: このモードはコスト超過時の中間オプション（§11 撤退ライン直前）として位置づけられているが、tags が空のままでは『グラフ層は動いているが検索品質は grep 以下』という状態が生まれ、撤退判断をさらに遅らせる。また逆輸入時に『LLM 抽出なしモードあり』と説明する際、成立条件（統制されたタグ語彙）が暗黙のままだと 組織展開 側で再現しない。

### [low] ULID の一意性・不変性を検証する CI チェックが未記載

**なぜ問題か**: エピソード名 {id}@{content_hash} の決定論化、superseded_by チェーン、plk_history の全てが id の一意性を前提にしている。重複が入ると『旧エピソード削除→再追加』の更新パスが別ファクトのエピソードを誤削除し、SoT（Git）は正しいのに派生索引が静かに壊れる。全再構築で直るため致命ではないが、原因特定に時間を食う類のバグ。

### [low] content_hash の正規化が未定義 — frontmatter ラウンドトリップの表記揺れで無内容変更でも hash が変わり、高コストな日本語 ingest が無駄に再走する

**視点**: consistency
